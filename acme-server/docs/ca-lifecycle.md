# Жизненный цикл CA: root, intermediate, бэкапы, доставка

Подробный гайд по правильной настройке двухуровневой CA-иерархии для secutor-acme: создание **offline root** и **online intermediate**, верификация цепочки, бэкапы с тестом восстановления, безопасное удаление root-ключа с рабочей машины, доставка intermediate на сервер ACME, ротация и отзыв.

Этот документ — обязательное чтение перед прод-запуском. CA — это фундамент доверия для всех сертификатов в твоей инфраструктуре. Ошибка в этом флоу не лечится «потом сделаем правильно»: придётся выпускать новый root, заново раздавать trust anchor всем клиентам, перевыпускать всё.

## Зачем root + intermediate

В простой схеме у тебя одна CA, она и подписывает клиентские сертификаты. Проблема: её приватный ключ постоянно нужен на сервере, чтобы что-то подписывать. Если сервер взломали — ключ украли — вся инфраструктура скомпрометирована, и **отзыв этой CA технически невозможен** (она сама себя подписала, доверять её отзыву некому).

Двухуровневая схема решает это:

- **Root CA** — самоподписанный, ключ держится **offline** (USB в сейфе, отдельная машина, бумажный бэкап). Подписывает только intermediate'ы. Появляется в живом мире несколько раз за свою жизнь.
- **Intermediate CA** — подписан root'ом, ключ живёт **online** на сервисной машине, обслуживает ACME. Если его взломали — root отозвал его, выпустил новый intermediate, клиенты по-прежнему доверяют root'у. Перевыпускать trust anchor у всех клиентов не надо.

Цена — один лишний сертификат в цепочке (`client → intermediate → root` вместо `client → root`). По функциональности эквивалентно, по безопасности — несравнимо лучше.

**Все настоящие CA так и делают**: Let's Encrypt → root ISRG Root X1 → intermediate R3 → твой сертификат. То же самое DigiCert, GlobalSign, Sectigo. Не изобретай велосипед, повторяй проверенную схему.

## Концепция: offline root и online intermediate

```
   ┌───────────────────────────────────┐
   │      Root CA (offline)             │
   │      ~/.secutor/contexts/root/     │
   │      ▸ store.enc (приватный ключ)  │
   │      ▸ cert.pem (публичный, root)  │
   │                                     │
   │      [USB в сейфе] [бэкап #2] [...] │
   └────────────┬───────────────────────┘
                │ подписывает (раз в годы)
                ▼
   ┌───────────────────────────────────┐
   │      Intermediate CA (online)      │
   │      ~/.secutor/contexts/inter./   │
   │      ▸ store.enc (приватный ключ)  │
   │      ▸ cert.pem (подписан root'ом) │
   └────────────┬───────────────────────┘
                │ подписывает (часто, через ACME)
                ▼
   ┌───────────────────────────────────┐
   │      Клиентские сертификаты        │
   │      app.lan.vpn, api.lan.vpn, ... │
   └───────────────────────────────────┘
```

**Контекст** в терминах secutor — это папка с зашифрованным bundle (`store.enc` + публичные данные + метаданные). Каждая CA = свой контекст = свой пароль шифрования.

## Шаг 1. Инициализация root CA

На **админской машине** (не на сервере, не на VPS — на твоём ноутбуке/рабочей станции):

```bash
secutor-cli init-ca \
  --name root \
  --common-name "Internal Networks Root CA" \
  --organization "Your Org" \
  --validity 20y \
  --key-type ed25519 \
  --password-prompt
# попросит ввести пароль для шифрования root context
```

Параметры:

- **`--name root`** — идентификатор контекста, будет создан `~/.secutor/contexts/root/`.
- **`--common-name`** — отображаемое имя, увидят пользователи в подробностях TLS-сертификата.
- **`--validity 20y`** — root живёт долго, обычно 15-25 лет. Менять root = заново раздавать trust anchor всем клиентам, операция болезненная.
- **`--key-type ed25519`** — современный алгоритм, маленькие ключи, быстрая верификация. Альтернатива — `rsa-4096` (для совместимости со старыми клиентами).
- **`--password-prompt`** — пароль вводишь интерактивно, не через флаги командной строки (иначе он попадёт в history).

> Точные имена флагов могут отличаться в твоей версии secutor-cli — сверь с `secutor-cli init-ca --help`.

**Пароль root** должен быть:
- длинным (passphrase из 5-7 слов из словаря, или 32+ случайных символов),
- уникальным (не использован больше нигде),
- сохранённым в менеджере паролей **и** на бумаге в сейфе.

После выполнения:

```bash
ls ~/.secutor/contexts/root/
# meta.json   cert.pem   store.enc
```

`cert.pem` — публичный сертификат root, можно куда угодно. `store.enc` — зашифрованный bundle с приватным ключом, охраняй как зеницу ока.

## Шаг 2. Инициализация intermediate CA

В той же сессии на админской машине:

```bash
secutor-cli init-intermediate \
  --root-context root \
  --root-password-prompt \
  --name intermediate \
  --common-name "Internal Networks Intermediate CA" \
  --organization "Your Org" \
  --validity 5y \
  --key-type ed25519 \
  --password-prompt
# попросит пароль root (для подписи), потом пароль для нового intermediate context
```

Параметры:

- **`--root-context root`** — указывает, какой контекст использовать для подписи (твой только что созданный root).
- **`--validity 5y`** — intermediate живёт меньше, обычно 3-10 лет. Чем короче, тем чаще ротация, но и тем выше безопасность.
- **`--password-prompt`** — отдельный пароль для intermediate context. **Не тот же**, что у root. Это второй слой защиты: если кто-то найдёт `store.enc` intermediate, он не подойдёт паролем от root.

После:

```bash
ls ~/.secutor/contexts/intermediate/
# meta.json   cert.pem   store.enc
```

`cert.pem` intermediate — это сертификат, подписанный root'ом. Это и есть «средняя ссылка» в цепочке.

## Шаг 3. Верификация цепочки

**Обязательный шаг.** Прежде чем что-то делать дальше, убедись, что подпись валидна:

```bash
openssl verify \
  -CAfile ~/.secutor/contexts/root/cert.pem \
  ~/.secutor/contexts/intermediate/cert.pem
# должно вывести: cert.pem: OK
```

Если `error 18 at 0 depth lookup: self signed certificate` или другая ошибка — что-то не так с подписью intermediate. Не двигайся дальше, пока не исправишь. **Особенно** не удаляй root — он понадобится для пересоздания intermediate.

Дополнительно — посмотри метаданные обоих:

```bash
openssl x509 -in ~/.secutor/contexts/root/cert.pem -text -noout | head -20
openssl x509 -in ~/.secutor/contexts/intermediate/cert.pem -text -noout | head -20
```

Проверь:
- **Issuer** intermediate'а = **Subject** root'а (это и означает «подписан root'ом»).
- У root **CA: TRUE** в Basic Constraints.
- У intermediate тоже **CA: TRUE** (иначе он не сможет подписывать клиентов).
- **Key Usage** у обоих включает `Certificate Sign` и `CRL Sign`.

## Шаг 4. Тестовый выпуск через intermediate

Прежде чем удалять root — проверь, что intermediate реально умеет подписывать клиентские сертификаты:

```bash
# выпуск тестового сертификата (синтаксис зависит от secutor-cli)
secutor-cli issue \
  --context intermediate \
  --password-prompt \
  --domain test.lan.vpn \
  --validity 7d \
  --out /tmp/test-cert.pem

# проверка полной цепочки до root
openssl verify \
  -CAfile ~/.secutor/contexts/root/cert.pem \
  -untrusted ~/.secutor/contexts/intermediate/cert.pem \
  /tmp/test-cert.pem
# должно: /tmp/test-cert.pem: OK
```

Если этот тест не прошёл — root **ещё нужен** на машине для отладки. Не удаляй.

Если прошёл — **удали** тестовый сертификат: `rm /tmp/test-cert.pem`. Он больше не нужен.

## Шаг 5. Бэкап root в несколько мест

Минимум **три копии** в разных физических местах. Это правило 3-2-1 из мира backup: три копии, два разных носителя, одна копия offsite.

### 5.1. Подготовка архива

```bash
cd ~/.secutor/contexts
tar czf /tmp/root-backup-$(date +%F).tgz root/
```

### 5.2. Двойное шифрование

`store.enc` внутри архива уже зашифрован паролем контекста, но накинем сверху ещё один слой GPG. Зачем — потому что в случае компрометации одного пароля у атакующего ещё одна стена впереди:

```bash
gpg --symmetric --cipher-algo AES256 /tmp/root-backup-$(date +%F).tgz
# попросит ввести GPG-пароль. Должен быть РАЗНЫЙ с паролем контекста.
# на выходе: /tmp/root-backup-YYYY-MM-DD.tgz.gpg
```

Оригинальный незашифрованный архив сразу убей:

```bash
shred -u /tmp/root-backup-$(date +%F).tgz
```

### 5.3. Распределение копий

| Копия | Носитель | Где хранить |
|---|---|---|
| #1 | USB-флешка (LUKS/encrypted) | Сейф в офисе |
| #2 | USB-флешка (LUKS/encrypted) | Сейф дома / другое физическое место |
| #3 | Шифрованное облако (S3 с SSE, или Tarsnap, или Borg к удалённому серверу) | offsite |

Дополнительно — **бумажный бэкап**. Распечатай:
- Серийный номер сертификата root.
- Fingerprint (SHA-256) сертификата.
- Пароль контекста.
- Пароль GPG.

Положи в сейф. Бумага переживает магнитные бури, EMP, разряды и забытые USB-флешки.

```bash
openssl x509 -in ~/.secutor/contexts/root/cert.pem -noout -serial -fingerprint -sha256
```

## Шаг 6. Тест восстановления бэкапа

**Самый важный шаг, который все пропускают. Бэкап без проверки = отсутствие бэкапа.**

На **другой машине** (или хотя бы в другой папке, в идеале — на чистой VM):

```bash
mkdir /tmp/restore-test && cd /tmp/restore-test

# скопируй бэкап с USB сюда
cp /media/usb-backup/root-backup-2026-05-28.tgz.gpg .

# расшифровка GPG
gpg --decrypt root-backup-2026-05-28.tgz.gpg > root-backup.tgz
# попросит GPG-пароль

# распаковка
tar xzf root-backup.tgz
ls root/
# meta.json   cert.pem   store.enc

# проверь, что store.enc открывается тем же паролем контекста
secutor-cli context info --dir ./root --password-prompt
# должна вывестись инфа о CA без ошибок: имя, валидность, fingerprint
```

Если **всё прошло** — отлично, бэкап валидный. Удали тестовую папку:

```bash
shred -u /tmp/restore-test/root/store.enc /tmp/restore-test/root-backup.tgz.gpg
rm -rf /tmp/restore-test
```

Если **хоть что-то сломалось** — **не удаляй оригинал**, разбирайся с бэкапом. Самые частые причины:
- Опечатался при сохранении пароля.
- Архив побился при копировании на USB (проверь `sha256sum` до и после).
- GPG-версия отличается между машинами (старый GPG не понимает новый алгоритм).
- Файловая система USB не поддерживает права (FAT32 теряет executable bit — не критично, но проверь).

**Повтори этот тест с каждой из трёх копий бэкапа.** Серьёзно. Это 15 минут вашего времени против «потеряли CA, перевыпускаем всё».

## Шаг 7. Сохранение root cert.pem для trust anchor

`cert.pem` из root context — это **публичная** часть, не секрет. Это и есть тот самый файл, который ты будешь раздавать клиентам как корень доверия.

Скопируй его в несколько удобных мест:

```bash
# в репозиторий документации (можно коммитить — это публичный сертификат)
cp ~/.secutor/contexts/root/cert.pem ~/repo/docs/root-ca.pem

# на сервисную машину, чтобы раздавать через HTTP
scp ~/.secutor/contexts/root/cert.pem \
    services-host:~/services/coredns/static/root.crt

# в трекер ассетов (Confluence/Wiki) как «trust anchor for internal CA»
```

Это нужно сделать **до** удаления root context, потому что после удаления cert.pem ты тоже потеряешь (или придётся доставать из бэкапа, что геморрой).

В отличие от приватного ключа, `cert.pem` может лежать публично где угодно — атакующий и так увидит его в каждом TLS-handshake. Шифровать его, прятать — бессмысленно.

## Шаг 8. Удаление root с рабочей машины

**Только после** того, как:

- [x] цепочка проверена (шаг 3),
- [x] тестовый выпуск прошёл (шаг 4),
- [x] три бэкапа сделаны (шаг 5),
- [x] бэкап **восстановлен и проверен** на тестовой машине (шаг 6),
- [x] root cert.pem скопирован в trust anchor хранилища (шаг 7).

Если хоть одна галочка не стоит — **не удаляй**. Возвращайся, доделай.

```bash
# удаляем приватный ключ root с перезаписью блоков
shred -u ~/.secutor/contexts/root/store.enc

# удаляем оставшиеся файлы
rm -rf ~/.secutor/contexts/root/
```

Использовать `shred -u` (а не просто `rm`) — чтобы блоки на диске были перезаписаны несколько раз, а не помечены свободными.

**Нюанс с SSD**: `shred` менее эффективен из-за wear leveling — ОС может физически писать перезаписи в другие блоки, а оригинальные данные останутся в неиспользуемых страницах. Для полной чистки SSD используй full-disk encryption и при компрометации/выводе из строя — смену ключа шифрования диска (LUKS `cryptsetup luksKillSlot`). Если диск SSD без FDE — root context лучше изначально создавать на encrypted volume.

## Шаг 9. Доставка intermediate на сервисную машину

Теперь у тебя есть проверенный intermediate, которому можно дать жить на сервере и подписывать клиентские сертификаты.

### 9.1. Копирование контекста

```bash
# с админской машины на сервисную
rsync -av --chmod=600 \
  ~/.secutor/contexts/intermediate/ \
  services-host:~/services/acme/context/

# проверь права
ssh services-host 'ls -la ~/services/acme/context/'
# все файлы должны быть -rw------- (600), владелец твой
```

### 9.2. Доставка пароля контекста

Пароль intermediate должен попасть на сервисную машину в файл `~/services/acme/secrets/context_password.txt`. **Не передавай его через bash-history**.

Лучший способ:

```bash
# на сервисной машине, интерактивно
ssh services-host
mkdir -p ~/services/acme/secrets
chmod 700 ~/services/acme/secrets
read -s -p "Intermediate password: " PWD
echo -n "$PWD" > ~/services/acme/secrets/context_password.txt
unset PWD
chmod 600 ~/services/acme/secrets/context_password.txt

# проверка: длина файла = длине пароля без \n в конце
wc -c ~/services/acme/secrets/context_password.txt
```

**Проверь**, что в конце нет `\n`:

```bash
xxd ~/services/acme/secrets/context_password.txt | tail -1
# последний байт должен быть последним символом пароля, не 0a
```

Альтернатива — Docker swarm secrets, Vault, sealed-secrets. См. [deployment.md](deployment.md), раздел про Docker secrets.

### 9.3. Запуск ACME

```bash
ssh services-host 'cd ~/services && docker compose up -d'
docker compose logs -f acme
```

В логах должно быть что-то вроде:

```
{"level":"info","msg":"loading context","name":"intermediate"}
{"level":"info","msg":"context unlocked successfully"}
{"level":"info","msg":"ACME server listening","addr":"0.0.0.0:8443"}
```

Если `failed to unlock context: invalid password` — проверь пароль (см. [dns-acme-peer.md, типичные грабли](dns-acme-peer.md#типичные-грабли)).

## Что остаётся на админской машине

После всего флоу:

```
~/.secutor/contexts/
└── intermediate/          ← оставляем, нужно для админских команд (revoke, info, rotate)
    ├── store.enc
    ├── cert.pem
    └── meta.json
```

**Root отсутствует**, лежит только в офлайн-бэкапах.

В менеджере паролей (или бумаге в сейфе):

- Пароль root context.
- Пароль intermediate context.
- Пароль GPG для архива.
- Fingerprint root cert.pem (чтобы при восстановлении сразу убедиться, что взял правильный).

## Что и где должно лежать после всего

| Файл/секрет | Админская машина | Сервисная машина | Офлайн-бэкап | Trust anchor хранилища |
|---|---|---|---|---|
| `root/store.enc` (приватный ключ root) | ❌ | ❌ | ✅ (×3) | ❌ |
| `root/cert.pem` (публичный сертификат root) | ✅ | ✅ (для раздачи) | ✅ | ✅ |
| `intermediate/store.enc` (приватный ключ intermediate) | ✅ | ✅ | ⚠️ опционально | ❌ |
| `intermediate/cert.pem` (публичный сертификат intermediate) | ✅ | ✅ | ✅ | ⚠️ опционально |
| Пароль root context | менеджер паролей + бумага | ❌ | ❌ | ❌ |
| Пароль intermediate context | менеджер паролей | ✅ в `secrets/` | ❌ | ❌ |
| Пароль GPG | менеджер паролей + бумага | ❌ | ❌ | ❌ |

## Когда снова нужен root

Root спит в сейфе. Но иногда его придётся будить:

1. **Intermediate скоро истечёт** (за 6-12 месяцев до конца срока) → надо выпустить новый intermediate, чтобы клиенты успели начать получать сертификаты от него до того, как старый умрёт.
2. **Intermediate скомпрометирован** → надо немедленно его отозвать, выпустить новый.
3. **Меняешь алгоритм** (например, переходишь с RSA на Ed25519) → новый intermediate, постепенный переход.
4. **Добавляешь второй intermediate** (например, отдельный для другой среды или роли) → подписываешь его root'ом.

Каждый из этих случаев — на свой день. См. ниже.

### Процедура «достать root для подписания»

1. Достаёшь из сейфа USB-флешку с бэкапом #1.
2. На **отдельной чистой машине** (живой Linux с USB, не на рабочей машине, точно не на сервере) — расшифровываешь GPG, распаковываешь архив.
3. Используешь root для подписания того, ради чего пришёл (новый intermediate, отзыв, что-то ещё).
4. **Сразу же** удаляешь все следы root с этой машины (`shred`, выключение, удаление volume на VM).
5. Возвращаешь USB-флешку обратно в сейф.

Не «оставлю-ка root на ноутбуке, вдруг ещё пригодится» — это анти-паттерн, ломающий весь смысл offline-схемы.

---

## Ротация intermediate

За 6-12 месяцев до истечения текущего intermediate:

1. Достаёшь root из бэкапа (см. выше).
2. Создаёшь **новый** intermediate с другим именем:
   ```bash
   secutor-cli init-intermediate \
     --root-context root \
     --name intermediate-2030 \
     --common-name "Internal Networks Intermediate CA 2030" \
     --validity 5y
   ```
3. Доставляешь новый intermediate на сервисную машину **параллельно** старому (secutor должен уметь поддерживать несколько активных подписывающих ключей).
4. Постепенно переводишь выпуск новых сертификатов на новый intermediate (через конфиг ACME-сервера).
5. Когда **все** клиентские сертификаты от старого intermediate истекли (или близки к этому) — удаляешь старый intermediate с сервера.
6. Прячешь root обратно.

Цель — чтобы у клиентов **всегда** была валидная цепочка к доверенному root'у, без window'а простоя.

---

## Компрометация и отзыв

Если есть подозрение, что **intermediate** скомпрометирован (украли `store.enc` + пароль):

1. Извлеки root из бэкапа.
2. Сгенерируй **CRL** (Certificate Revocation List), куда добавь сертификат скомпрометированного intermediate:
   ```bash
   secutor-cli revoke-intermediate \
     --root-context root \
     --target-cert ~/.secutor/contexts/intermediate/cert.pem \
     --reason key-compromise \
     --out /tmp/crl.pem
   ```
3. Опубликуй CRL по известному URL (`http://acme.lan.vpn/crl.pem`). У клиентов в trust store должна быть настроена проверка CRL (на практике почти никто не настраивает, так что готовься к ручной чистке).
4. Выпусти новый intermediate с другим ключом.
5. Доставь на сервер.
6. Старый intermediate **немедленно** убери из конфига ACME.
7. Закрытым каналом (письма, чат) уведоми пользователей, что нужно проверить, не получали ли они подозрительных сертификатов от старого intermediate.

Если скомпрометирован **root** — это катастрофа:

1. Выпуск нового root.
2. Раздача нового trust anchor **всем** клиентам — физически, по защищённому каналу.
3. Перевыпуск всего: новый intermediate, все клиентские сертификаты.
4. Старый root и всё, что им подписано, объявляются недоверенными (если у кого-то по ошибке остался — отзывать всё равно бесполезно, key compromise root неотменяем).

Поэтому root и хранится offline. Чтобы такого не случалось.

## Доставка trust anchor клиентам

Клиенты должны доверять **root** (не intermediate). Логика — root стабилен, intermediate ротируешь без надобности что-то пересоздавать у клиентов.

### Способ 1: HTTP-раздача с сервисной машины

В compose из [dns-acme-peer.md](dns-acme-peer.md) добавь маленький nginx (или используй фичу secutor-acme, если есть):

```yaml
  static:
    image: nginx:alpine
    network_mode: "service:wg"
    depends_on: [wg]
    volumes:
      - ./static:/usr/share/nginx/html:ro
    restart: unless-stopped
```

И положи root cert в `./static/root.crt`. Будет доступен по `http://acme.lan.vpn/root.crt`.

### Способ 2: Через configuration management

Ansible/Salt/Chef деплоит `root.crt` в `/usr/local/share/ca-certificates/` на каждой Linux-машине и запускает `update-ca-certificates`. Для macOS — через MDM (Jamf, Intune). Для Windows — через GPO.

### Способ 3: Вручную (для маленьких сетей)

В [dns-acme-peer.md, шаг 11](dns-acme-peer.md#шаг-11-регистрация-trust-anchor-у-клиентов) — пошагово для Linux/macOS/Windows.

### Что НЕ доставлять клиентам

- **Не** доставляй intermediate cert как trust anchor. Тогда при ротации intermediate'а нужно будет менять trust у всех. Бессмысленный геморрой.
- **Не** доставляй приватный ключ root никому ни при каких обстоятельствах.

## Типичные ошибки

### «Удалил root после бэкапа, но не проверил бэкап»

Самая частая катастрофа. Через год понадобился root (для нового intermediate), достаёшь USB — а пароль не подходит/архив побился/GPG не открывается. Поздно.

**Лечение**: шаг 6 обязателен. Серьёзно.

### «Сохранил пароль контекста в `init-ca` через флаг `--password=xxx`»

Пароль попадает в `ps`, `history`, логи. **Лечение**: только `--password-prompt` или `--password-file /run/secrets/...`.

### «Положил root context на тот же диск, что и intermediate»

Если кто-то получит доступ к диску — получит оба. **Лечение**: root всегда на отдельном offline-носителе.

### «Не сохранил root cert.pem перед удалением context»

Пришлось доставать из бэкапа только ради публичной части (которую можно было разложить везде). **Лечение**: шаг 7 — обязательно копируй cert.pem в несколько мест **до** удаления.

### «Доставил intermediate и пароль в одном `scp` командой со всеми переменными в bash-history»

Пароль навсегда в history. **Лечение**: пароль вводить интерактивно через `read -s`, не через переменные окружения и не через флаги.

### «Доверяю intermediate напрямую вместо root»

Через пару лет надо ротировать intermediate — а у всех клиентов в trust store именно он. Невозможно ротировать без массового переустановления trust. **Лечение**: trust anchor — это root, всегда.

### «Один пароль на root и intermediate»

Скомпрометировали store.enc intermediate → подобрали пароль (атакой по словарю или утечкой) → попробовали этот же пароль на root → готово. **Лечение**: разные пароли, длинные, в менеджере паролей.

### «Не настроил уведомления об истечении intermediate»

Истёк через 5 лет неожиданно, все сервисы упали. **Лечение**: алерт за 90 дней до истечения (`certbot renew --dry-run`, или скрипт-проверка валидности через cron, или Prometheus blackbox-exporter).

## Чек-лист

Прежде чем считать установку завершённой:

- [ ] Root CA создан с паролем, валидность 15-25 лет, ключ ed25519 или RSA-4096.
- [ ] Intermediate CA создан с **другим** паролем, валидность 3-10 лет, подписан root'ом.
- [ ] `openssl verify -CAfile root.pem intermediate.pem` → OK.
- [ ] Тестовый клиентский сертификат выпустился через intermediate, цепочка проверена до root.
- [ ] Сделаны **три** бэкапа root, зашифрованы GPG поверх внутреннего шифрования.
- [ ] **Тест восстановления** прошёл успешно на отдельной машине (не на той, где делали бэкап).
- [ ] Пароли (root, intermediate, GPG) сохранены в менеджере паролей **и** на бумаге в сейфе.
- [ ] Fingerprint root cert.pem распечатан на бумаге.
- [ ] Root cert.pem скопирован в trust anchor хранилища (репозиторий docs, сервисная машина для раздачи, etc.).
- [ ] Root context **удалён** с рабочей машины через `shred -u`.
- [ ] Intermediate context доставлен на сервисную машину с правами `600`.
- [ ] Пароль intermediate доставлен **интерактивно**, без `\n` в конце файла, права `600`.
- [ ] ACME-сервер успешно стартует и видит intermediate (`context unlocked` в логах).
- [ ] Trust anchor (root cert) установлен на админских машинах, тестовых клиентах.
- [ ] Алерт на истечение intermediate настроен (срабатывает за 90 дней).
- [ ] Документирована процедура «достать root из бэкапа» — где лежит, под каким паролем, на чём расшифровать.
- [ ] Документирована процедура ротации intermediate.
