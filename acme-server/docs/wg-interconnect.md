## Зачем это нужно

В контексте ACME-сервера и LAN-сертификатов почти всегда нужна одна из этих историй:

1. **Один хаб, много клиентов.** На хабе живёт CA, клиенты приходят за сертификатами. Между собой клиентам обычно общаться не надо — но иногда надо (например, мониторинг с одного клиента должен скрейпить экспортёры на других).
2. **Site-to-site.** На клиенте за WG стоит целая локальная сеть (`192.168.20.0/24`), и тебе надо, чтобы хосты из VPN видели машины в этой локалке как будто они в общей сети.
3. **Несколько площадок, прямые туннели между ними.** Хочется, чтобы трафик между двумя датацентрами не заворачивался на хаб, а шёл напрямую.
4. **Слияние двух независимых WG-сетей.** Раньше были два отдельных VPN, теперь надо, чтобы клиенты одного видели клиентов другого.

Все четыре сценария — это одна и та же механика WG плюс правильно настроенная маршрутизация и (иногда) NAT.

## Как WireGuard думает про маршруты

В WireGuard нет понятия «сервер» и «клиент». Есть только **пиры** (peers). У каждого пира есть пара ключей и список других пиров. Связь между ними определяется двумя полями:

- **`AllowedIPs`** в секции `[Peer]` — это одновременно:
  - **на входе**: фильтр. Пакет из туннеля принимается, только если его source IP входит в `AllowedIPs` этого пира. Всё остальное молча дропается.
  - **на выходе**: таблица маршрутизации. Когда WG поднимает интерфейс, он добавляет в системную routing table маршруты на все подсети из `AllowedIPs` через свой `wgX`.

  То есть `AllowedIPs` — это **роль пира в сети**: какие адреса он имеет право отправлять и куда ему слать пакеты. Это самая важная концепция, большинство «не работает» — из-за неё.

- **`Endpoint`** — публичный IP:порт удалённого пира. Нужен только тому, кто инициирует соединение. После handshake WG запоминает реальный адрес и обновляет его автоматически, так что для пиров за NAT он не указывается вовсе.

Дополнительные кусочки, которые делают возможным сложные топологии:

- **`net.ipv4.ip_forward=1`** — без этого Linux дропает пакеты, не предназначенные ему лично. На любом узле, который прокидывает трафик между двумя интерфейсами (между `wg0` и `eth0`, между `wg0` и `wg1`), это обязательно.
- **`iptables -A FORWARD ... -j ACCEPT`** — даже с `ip_forward=1` фаервол может не пускать. В современных дистрибутивах политика FORWARD часто DROP по умолчанию.
- **`iptables -t nat -A POSTROUTING ... -j MASQUERADE`** — подменяет source IP исходящих пакетов на адрес выходного интерфейса. Нужно, когда на удалённой стороне нет маршрута обратно к источнику.

Запомни эти три ингредиента — дальше всё крутится вокруг них.

## Топология 1. Hub-and-spoke + связь клиентов между собой

Самая частая схема. Один центральный пир (хаб) с публичным IP, все остальные — клиенты, которые подключаются к нему. Все клиенты в одной подсети `10.10.0.0/24`.

```
                    hub (10.10.0.1)
                    publicly reachable
                    ┌─────────────┐
                    │     wg0     │
                    └──┬───┬───┬──┘
                       │   │   │
              ┌────────┘   │   └────────┐
              │            │            │
          ┌───▼───┐    ┌───▼───┐    ┌───▼───┐
          │ peerA │    │ peerB │    │ peerC │
          │ .0.2  │    │ .0.3  │    │ .0.4  │
          └───────┘    └───────┘    └───────┘
```

### Шаг 1. Конфиг хаба

```ini
# /etc/wireguard/wg0.conf на хабе
[Interface]
Address    = 10.10.0.1/24
ListenPort = 51820
PrivateKey = <hub_priv>

# Включаем форвардинг и разрешаем трафик из wg0 обратно в wg0.
# Без этих правил клиенты будут видеть только хаб, друг друга — нет.
PostUp   = sysctl -w net.ipv4.ip_forward=1
PostUp   = iptables -A FORWARD -i %i -o %i -j ACCEPT
PostDown = iptables -D FORWARD -i %i -o %i -j ACCEPT

[Peer]
# peerA
PublicKey  = <A_pub>
AllowedIPs = 10.10.0.2/32

[Peer]
# peerB
PublicKey  = <B_pub>
AllowedIPs = 10.10.0.3/32

[Peer]
# peerC
PublicKey  = <C_pub>
AllowedIPs = 10.10.0.4/32
```

Обрати внимание: у каждого пира в `AllowedIPs` стоит **`/32`** (один конкретный адрес). Если поставить там `10.10.0.0/24`, WG не сможет понять, кому из пиров слать пакет на `10.10.0.5` — все три «имеют право».

### Шаг 2. Конфиг клиента A

```ini
# /etc/wireguard/wg0.conf на peerA
[Interface]
Address    = 10.10.0.2/32
PrivateKey = <A_priv>

[Peer]
PublicKey  = <hub_pub>
Endpoint   = hub.example.com:51820
AllowedIPs = 10.10.0.0/24            # !!! всю подсеть, а не только хаб
PersistentKeepalive = 25
```

**Ключевое отличие от хаба**: у клиента в `AllowedIPs` указана **вся подсеть VPN**, а не только адрес хаба. Это говорит системе: «весь трафик в `10.10.0.0/24` шли в `wg0`, дальше хаб разберётся». Если поставить `10.10.0.1/32` — клиент будет ходить только до хаба, до других клиентов не достучится никогда.

Симметрично настраиваешь peerB и peerC.

### Что в итоге работает

- A пингует хаб — да.
- A пингует B — да, через хаб (увидишь это в `traceroute`: один лишний хоп через `10.10.0.1`).
- B стучится в ACME на хабе по `https://acme.lan:8443` — да.

### Минусы

- Любая коммуникация между клиентами идёт через хаб → лишняя латентность, хаб — узкое место.
- Если хаб упал, клиенты теряют друг друга.
- Двойное шифрование? Нет: пакет дешифруется на хабе, форвардится в plain text внутри `wg0` namespace, потом шифруется обратно. Это **не двойное** шифрование, но всё равно лишний CPU на хабе.

## Топология 2. Site-to-site: объединение целых локалок

Сценарий: за peerB стоит офисная локалка `192.168.20.0/24`. Тебе надо, чтобы из VPN был доступ к принтерам, NAS, etc., которые сами WG не запускают.

```
   hub (10.10.0.1)
        │
        │ wg0  10.10.0.0/24
        │
   ┌────▼─────┐
   │  peerB    │  10.10.0.3
   │ (router)  │
   └────┬──────┘
        │ eth0 192.168.20.1
        │
   ┌────▼──────────────────────────┐
   │ офисная LAN  192.168.20.0/24   │
   │   ┌──────┐  ┌──────┐  ┌─────┐ │
   │   │ NAS  │  │print │  │ PC  │ │
   │   └──────┘  └──────┘  └─────┘ │
   └────────────────────────────────┘
```

### Шаг 1. На хабе говорим, что подсеть B живёт за peerB

```ini
[Peer]
# peerB — теперь это не просто клиент, а гейтвей в офисную сеть
PublicKey  = <B_pub>
AllowedIPs = 10.10.0.3/32, 192.168.20.0/24
```

Это и есть ключевая магия `AllowedIPs`: хаб теперь знает, что любой пакет на `192.168.20.*` нужно зашифровать ключом peerB и отправить в его туннель.

### Шаг 2. На peerB включаем форвардинг и NAT

```ini
[Interface]
Address    = 10.10.0.3/32
PrivateKey = <B_priv>

PostUp   = sysctl -w net.ipv4.ip_forward=1
# разрешаем форвардинг между wg0 и eth0 в обе стороны
PostUp   = iptables -A FORWARD -i %i -o eth0 -j ACCEPT
PostUp   = iptables -A FORWARD -i eth0 -o %i -j ACCEPT
# подменяем источник на свой адрес в локалке (см. примечание ниже)
PostUp   = iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE

PostDown = iptables -D FORWARD -i %i -o eth0 -j ACCEPT
PostDown = iptables -D FORWARD -i eth0 -o %i -j ACCEPT
PostDown = iptables -t nat -D POSTROUTING -o eth0 -j MASQUERADE

[Peer]
PublicKey  = <hub_pub>
Endpoint   = hub.example.com:51820
AllowedIPs = 10.10.0.0/24
PersistentKeepalive = 25
```

### Шаг 3. На клиенте A добавляем маршрут к подсети B

```ini
[Peer]
PublicKey  = <hub_pub>
Endpoint   = hub.example.com:51820
AllowedIPs = 10.10.0.0/24, 192.168.20.0/24
PersistentKeepalive = 25
```

Теперь A знает: «трафик на 192.168.20.* — в `wg0`, а там хаб разберётся (а хаб знает, что это к peerB)».

### Про MASQUERADE — обязателен или нет?

Зависит от того, есть ли в локалке `192.168.20.0/24` обратный маршрут к VPN-подсети `10.10.0.0/24`:

- **MASQUERADE есть**: NAS видит запрос как пришедший от `192.168.20.1` (адрес peerB в локалке). Ответ возвращается peerB естественно, без дополнительных маршрутов. Минус — на стороне NAS все клиенты VPN выглядят одинаково (логи бесполезны для аудита).
- **MASQUERADE нет**: NAS видит реальный source IP `10.10.0.2`. Чтобы он мог ответить, на самом NAS (или на роутере локалки, если он default gateway для NAS) нужен маршрут:
  ```
  ip route add 10.10.0.0/24 via 192.168.20.1
  ```
  Если такой маршрут добавить можно — это **правильный** способ. Логи на NAS остаются информативными, аудит работает.

Для начала ставь MASQUERADE — это «всегда работает». Когда заматереешь — убирай и прописывай маршруты.

### Подводный камень: пересечение подсетей

Если в VPN у тебя `192.168.1.0/24`, а у клиента A дома такая же сеть — ловушка. Локальный default route выиграет, и трафик никогда не уйдёт в туннель. **Всегда выбирай экзотические подсети для WG**: `10.99.x.x`, `10.42.x.x`, `172.30.x.x` — что-то, с чем заведомо не пересечётся ни одна домашняя сеть.

## Топология 3. Full mesh: каждый с каждым

В hub-and-spoke хаб видит весь трафик и добавляет латентность. В меше каждый пир знает каждого и ходит напрямую.

```
       peerA ─────────── peerB
         │   ╲         ╱   │
         │     ╲     ╱     │
         │       ╲ ╱       │
         │       ╱ ╲       │
         │     ╱     ╲     │
         │   ╱         ╲   │
       peerD ─────────── peerC
```

### Конфиг каждого пира

У каждого пира — N−1 секций `[Peer]`, по одной на каждого другого. У peerA:

```ini
[Interface]
Address    = 10.10.0.2/32
ListenPort = 51820
PrivateKey = <A_priv>

[Peer]
PublicKey  = <B_pub>
Endpoint   = peerB.example.com:51820
AllowedIPs = 10.10.0.3/32
PersistentKeepalive = 25

[Peer]
PublicKey  = <C_pub>
Endpoint   = peerC.example.com:51820
AllowedIPs = 10.10.0.4/32
PersistentKeepalive = 25

[Peer]
PublicKey  = <D_pub>
Endpoint   = peerD.example.com:51820
AllowedIPs = 10.10.0.5/32
PersistentKeepalive = 25
```

### Ограничения

- **Каждому пиру нужен достижимый Endpoint.** Если кто-то за NAT без проброса портов и без публичного IP — он не сможет принять входящее соединение. Меш с пирами за NAT работает только если хотя бы одна сторона достижима.
- **Конфиги растут квадратично.** На 5 пирах руками — терпимо. На 20 — мучение, изменения тоже мучительные (добавил одного — правишь все).

### Что использовать вместо ручной правки

- **[wg-meshconf](https://github.com/k4yt3x/wg-meshconf)** — генератор конфигов из одного YAML.
- **[Netmaker](https://www.netmaker.io/)** — control plane для WG, сам раздаёт конфиги и умеет в NAT traversal.
- **[Tailscale](https://tailscale.com/)** / **[Headscale](https://github.com/juanfont/headscale)** (self-hosted) — не чистый WG, но на нём построено. Решают проблему NAT через DERP-релеи.
- **[Nebula](https://github.com/slackhq/nebula)** — не WG, идейно похоже, изначально заточено под меш.

Если в проекте уже стоит ACME-сервер с интеграцией DNS — Headscale прекрасно ложится рядом.

## Топология 4. Соединение двух независимых WG-сетей

Бывает ситуация: исторически на площадке X собрана своя WG-сеть `10.10.0.0/24`, на площадке Y — своя `10.20.0.0/24`, у каждой свой хаб. Теперь хочется, чтобы клиент из X видел клиента из Y и наоборот.

Решение: между хабами поднимаешь **отдельный** WG-туннель (назовём его `wg2`), и через него хабы маршрутизируют трафик в чужие подсети.

```
    wg0: 10.10.0.0/24         wg0: 10.20.0.0/24
    ┌──────────┐              ┌──────────┐
    │  hub X   │              │  hub Y   │
    │ 10.10.0.1│              │ 10.20.0.1│
    │          │  wg2 link    │          │
    │          ├──────────────┤          │
    │ 10.99.0.1│ 10.99.0.0/30 │ 10.99.0.2│
    └────┬─────┘              └────┬─────┘
         │                         │
    peerA(.10)                 peerY(.10)
```

### На хабе X: два конфига интерфейсов

`wg0.conf` — как раньше, обслуживает локальных клиентов X. Дополнительно в нём добавляем маршрут на чужую подсеть через `wg2`:

```ini
[Interface]
Address    = 10.10.0.1/24
ListenPort = 51820
PrivateKey = <hubX_priv>
PostUp     = sysctl -w net.ipv4.ip_forward=1
PostUp     = iptables -A FORWARD -i %i -o %i -j ACCEPT
PostUp     = iptables -A FORWARD -i %i -o wg2 -j ACCEPT
PostUp     = iptables -A FORWARD -i wg2 -o %i -j ACCEPT

[Peer]
# peerA из сети X
PublicKey  = <A_pub>
AllowedIPs = 10.10.0.10/32
```

`wg2.conf` — линк между хабами:

```ini
[Interface]
Address    = 10.99.0.1/30
ListenPort = 51821
PrivateKey = <hubX_link_priv>

[Peer]
# хаб Y
PublicKey  = <hubY_link_pub>
Endpoint   = hubY.example.com:51821
AllowedIPs = 10.99.0.2/32, 10.20.0.0/24    # чужой линк-IP + чужая подсеть
PersistentKeepalive = 25
```

Симметрично на хабе Y. Линк-подсеть `/30` потому, что в ней нужно ровно два адреса.

### На клиентах X: расширяем AllowedIPs

```ini
[Peer]
PublicKey  = <hubX_pub>
Endpoint   = hubX.example.com:51820
AllowedIPs = 10.10.0.0/24, 10.20.0.0/24    # своя сеть + чужая
PersistentKeepalive = 25
```

Теперь peerA посылает пакет на `10.20.0.10` → попадает в `wg0` к хабу X → хаб X видит маршрут «10.20.0.0/24 через wg2» → шифрует для хаба Y → хаб Y расшифровывает, видит, что цель `10.20.0.10` — это его клиент peerY → шифрует ключом peerY → доставляет.

### Альтернатива: вместо линка — peerхаб как клиент чужого хаба

Можно вместо отдельного `wg2` сделать так: хаб Y сам становится клиентом хаба X в его `wg0`. Технически работает, но смешивает роли — на одном интерфейсе и пользовательский трафик, и interlink. С точки зрения отладки и фаервол-правил отдельный `wg2` чище.

## NAT, MASQUERADE и обратные маршруты

Это самая частая причина «всё настроил, не работает». Краткое объяснение, когда что нужно.

### Когда NAT не нужен

Пакет идёт между пирами целиком в WG-пространстве: `10.10.0.2 → 10.10.0.3`. Ответ так же. Никакого NAT не нужно — WG сам знает, кому что слать.

### Когда нужен FORWARD

Пакет приходит в `wg0` и должен выйти в `eth0` (или в другой `wgX`). Линукс по умолчанию не пакеты транзитом не пускает, пока не разрешишь явно:

```bash
iptables -A FORWARD -i wg0 -o eth0 -j ACCEPT
iptables -A FORWARD -i eth0 -o wg0 -j ACCEPT
sysctl -w net.ipv4.ip_forward=1
```

И не забудь сохранить (`iptables-save`, systemd-unit, или PostUp/PostDown в конфиге WG).

### Когда нужен MASQUERADE

Когда **на принимающей стороне нет обратного маршрута** к источнику. Типовые случаи:

- В офисной локалке NAS не знает, что `10.10.0.0/24` существует и куда слать ответ.
- Пакет уходит в публичный интернет (источник — VPN-IP, но на той стороне его не существует).

MASQUERADE подменяет source на адрес выходного интерфейса, и conntrack потом возвращает ответ правильному получателю.

```bash
iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE
```

### Когда MASQUERADE НЕ нужен

Когда у обеих сторон есть взаимные маршруты — например, ты заранее настроил роутер офиса так, что он знает «10.10.0.0/24 живёт за peerB». Тогда лучше **не маскарадить** — сохранишь видимость реальных IP, аудит и фаервол-правила становятся осмысленными.

## MTU и фрагментация

WireGuard добавляет к каждому пакету заголовок ~60 байт. Если стандартный MTU интерфейса 1500, то внутри туннеля влезает 1420 (IPv4) или 1400 (IPv6) — WG сам так выставляет MTU для `wg0` по умолчанию.

Проблема: если поверх WG идёт ещё один туннель (например, GRE или второй VPN), или провайдер использует PPPoE (MTU 1492 вместо 1500), фактический MTU может оказаться меньше, и крупные пакеты будут молча теряться. Симптомы:

- ping маленькими пакетами работает, большими — нет.
- TCP-соединение устанавливается, но HTTPS-страница не догружается.
- SSH-сессия открывается, но висит при `ls` большой директории.

Что делать:

1. Прогнать `ping -M do -s 1400 <peer>` — постепенно уменьшая размер, найти максимум без фрагментации. Минус 28 байт (IP+ICMP заголовки) — это и есть рабочий MTU.
2. Прописать в `[Interface]`:
   ```ini
   MTU = 1380
   ```
3. Альтернатива (грубо, но работает): clamp TCP MSS на интерфейсе.
   ```bash
   iptables -t mangle -A FORWARD -o wg0 -p tcp --tcp-flags SYN,RST SYN -j TCPMSS --clamp-mss-to-pmtu
   ```

В чисто WG-меше без двойных туннелей дефолтный MTU обычно работает. Но в site-to-site через провайдеров проверь обязательно.

## DNS в объединённых сетях

Если в одной из объединённых сетей живёт DNS-сервер (CoreDNS/BIND), отдающий внутренние имена типа `acme.lan`, нужно, чтобы остальные пиры знали, что DNS живёт по этому адресу.

Два пути:

1. **Push DNS через `[Interface] DNS = ...`.** WG поднимает `wg-quick`, который через `resolvconf` или systemd-resolved прописывает указанный DNS как резолвер на стороне клиента, пока туннель активен.
   ```ini
   [Interface]
   Address = 10.10.0.2/32
   DNS     = 10.10.0.1
   PrivateKey = ...
   ```
   Чисто и удобно. Минус — `wg-quick` требует поддержки `resolvconf` или активного systemd-resolved.

2. **Split-DNS на хосте.** В `/etc/systemd/resolved.conf.d/wg.conf`:
   ```
   [Resolve]
   DNS=10.10.0.1
   Domains=~lan ~vpn
   ```
   Тогда `*.lan` и `*.vpn` идут только в VPN-резолвер, всё остальное — куда настроено по умолчанию. Это правильный подход для постоянно работающих машин (серверов, рабочих станций).

В docker-контейнерах удобнее всего пробросить DNS через `--dns 10.10.0.1` или в `docker-compose.yml`:
```yaml
services:
  app:
    dns: [10.10.0.1]
```

## Безопасность и ограничение доступа

WG сам по себе шифрует трафик, но **не делает контроль доступа** между пирами. Если пирам можно дотянуться друг до друга по IP — они дотянутся. Если тебе нужно, чтобы peerA ходил **только** в ACME на хабе и больше никуда:

1. **На хабе** добавь фаервол на FORWARD:
   ```bash
   # разрешаем только трафик от peerA на хаб (8443)
   iptables -A FORWARD -i wg0 -s 10.10.0.2 -d 10.10.0.1 -p tcp --dport 8443 -j ACCEPT
   # дропаем всё остальное от peerA
   iptables -A FORWARD -i wg0 -s 10.10.0.2 -j DROP
   ```

2. **На клиенте** сузь `AllowedIPs`:
   ```ini
   AllowedIPs = 10.10.0.1/32    # только хаб, других клиентов даже маршрут не построится
   ```

Второй способ — это **доверие со стороны клиента**: ты ему доверяешь, что он сам ограничит свой трафик. Первый — централизованный, не зависит от добросовестности клиентов. В проде делай оба.

### Pre-shared key

Дополнительный слой к публичному ключу. Просто симметричный 256-бит секрет, добавляется в обе стороны:

```ini
[Peer]
PublicKey    = ...
PresharedKey = <pSK base64>
```

Технически защищает от пост-квантового взлома публичной криптографии (если когда-нибудь). Практически — почти бесплатная страховка, делать стоит.

## Диагностика. Что делать, если не работает

Идти по слоям снизу вверх.

### 1. Handshake вообще есть?

```bash
sudo wg show
```

Смотри `latest handshake`:
- `(none)` или старше пары минут → туннель не установился. Проверяй `Endpoint`, `PublicKey`, фаервол на UDP-порту 51820 (или какой настроен).
- свежий (несколько секунд назад) → WG работает, проблема выше.

### 2. Пакеты идут через туннель?

```bash
sudo tcpdump -i wg0 -n
```

В другой консоли — `ping <peer>`. Видишь ICMP в обе стороны → ОК. Только в одну → проблема с обратным маршрутом или MASQUERADE.

### 3. ip_forward и FORWARD-правила

```bash
sysctl net.ipv4.ip_forward          # должно быть = 1
iptables -L FORWARD -v -n           # счётчики растут при попытках?
iptables -t nat -L POSTROUTING -v -n
```

Если счётчики на нужном правиле не растут — пакет до него не доходит, ищи раньше.

### 4. AllowedIPs

Самая частая ошибка. Проверь:
- У хаба `AllowedIPs` каждого клиента — это `/32` (или подсеть, которую тот клиент представляет).
- У клиента `AllowedIPs` хаба — это **вся VPN-подсеть** плюс все подсети, до которых надо ходить.
- Пересечений нет: один и тот же `/32` не указан у двух пиров.

### 5. Маршруты в системе

```bash
ip route get 10.10.0.5         # видит ли система, что это через wg0?
ip route get 192.168.20.10
```

Если `unreachable` или уходит не туда — `AllowedIPs` на этом узле неправильный, или есть конфликтующий маршрут с более узкой маской.

### 6. Conntrack для NAT

```bash
sudo conntrack -L | grep 192.168.20.10
```

Видишь записи с правильной подменой source — MASQUERADE работает.

### 7. Чужая локалка не знает обратного маршрута

Симптом: с VPN пинг до `192.168.20.10` не возвращается, на peerB `tcpdump -i eth0` показывает уходящий ICMP request, но ответ не приходит.

Лечение: MASQUERADE на peerB **или** статический маршрут на роутере локалки.

## Чек-лист перед прод-запуском

- [ ] У хаба `net.ipv4.ip_forward=1`, и это переживёт ребут (`/etc/sysctl.d/99-wg.conf`).
- [ ] PostUp/PostDown сохраняют состояние фаервола, а не падают молча. Проверь `journalctl -u wg-quick@wg0`.
- [ ] Все `AllowedIPs` асимметрично сужены: на хабе — `/32` (или конкретные подсети), у клиентов — вся VPN-подсеть.
- [ ] Подсеть VPN не пересекается ни с одной локальной сетью клиентов.
- [ ] `PersistentKeepalive = 25` стоит на стороне клиентов за NAT.
- [ ] Pre-shared keys выпущены и доставлены безопасным каналом.
- [ ] MTU проверен (`ping -M do -s 1400`), при необходимости занижен.
- [ ] DNS внутренних имён работает у всех клиентов (`getent hosts acme.lan`).
- [ ] Фаервол на хабе ограничивает доступ клиентов друг к другу, если они должны быть изолированы.
- [ ] Мониторинг: handshake age экспортируется в Prometheus или хотя бы крон-скрипт пишет в лог.
- [ ] Бэкап ключей хаба и `wg0.conf` лежит в защищённом месте. Потеря приватного ключа = всем клиентам новые конфиги.
